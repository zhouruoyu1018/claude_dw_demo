# 数仓需求拆解 (DW Requirement Triage)

从混合型业务需求文档、交互图（原型图/Wireframe/Mockup）和 Excel 导出模板中精准识别数据仓库开发范畴的内容，输出结构化的数仓需求清单。

## 输入类型

本 skill 支持三类输入，可单独使用或任意组合：

| 输入类型 | 格式 | 说明 |
|---------|------|------|
| **文本文档** | PRD/BRD 文本、Markdown、复制粘贴 | 传统需求文档 |
| **交互图** | 截图(PNG/JPG)、Figma 导出图、原型图照片 | 产品提供的 UI 设计稿 |
| **Excel 导出模板** | .xlsx/.xls 文件、Excel 截图 | 产品提供的报表导出样例模板 |

> 当多类输入同时存在时，执行多源交叉验证（见 Step 2）。

## 工作流程

```
输入(文档+交互图+Excel模板) → 交互图解析 → Excel模板解析 → 结构化扫描 → 多源交叉验证 → 语义级过滤 → 埋点识别 → 分层/引擎建议 → 输出结构化需求 → 持久化到 docs/wip/
```

### Step 0/0.1: 输入解析（如有交互图或 Excel 模板）

从交互图中提取筛选器→维度、列头→指标、图表轴→粒度等；从 Excel 模板中提取列头→字段、公式→计算逻辑、合计行→聚合方式。详见 [input-parsing.md](references/input-parsing.md)。

### Step 1: 结构化区域扫描

扫描文档目录或大标题，优先提取以下章节：
- "统计需求"、"报表需求"、"数据分析"
- "数据埋点"、"埋点设计"
- "非功能性需求-数据部分"
- "Dashboard"、"大屏"、"BI"

### Step 2: 多源交叉验证（多类输入同时存在时）

当同时存在文本文档、交互图、Excel 模板中的两种或以上时，执行交叉验证以提升需求完整性和准确度。

**验证矩阵**:

| 场景 | 处理方式 | 确信度 |
|------|---------|--------|
| **多源一致**: 文本/交互图/Excel 模板描述吻合 | 直接确认，标记为高确信度需求 | 高 |
| **图有文无**: 交互图中有筛选器/列头，但文本未提及 | 补充到需求清单，标注"来源：交互图，文本未描述" | 中 |
| **Excel有文无**: Excel 模板有此列，但文本未提及 | 补充到需求清单，标注"来源：Excel模板，文本未描述" | 中 |
| **文有图/Excel无**: 文本描述了指标，但交互图和 Excel 模板中均无 | 保留需求，标注"仅文本描述，可能是后续迭代" | 中 |
| **多源冲突**: 不同来源对同一字段的定义不一致 | 标记为待确认项，列出冲突细节 | 低 |

**常见补全场景**（产品经理容易遗漏但会画在图上或写在 Excel 中）:
- 筛选器维度、排序方式、分页提示、小计/合计行
- 精确字段名和列顺序、数据格式、公式暴露的口径、聚合维度

**Excel 与交互图互补**: 同一需求若同时有交互图和 Excel 模板，以 Excel 模板的字段定义为准（更精确），以交互图的筛选/下钻为补充。

### Step 3: 语义级过滤

按段落进行 OLTP vs OLAP 特征识别：

| 特征维度 | 应用开发 (排除) | 数仓开发 (保留) |
|---------|----------------|----------------|
| **关键词** | 录入、修改、点击、跳转、校验、弹窗 | 统计、分析、汇总、监控、下钻、趋势、环比、同比 |
| **时间维度** | 实时、当前状态 | 历史快照、T+1、周期性（每日/每月） |
| **数据形态** | 单条记录、事务性、明细操作 | 聚合数据、指标、维度 |
| **输出形式** | 页面跳转、弹窗、API响应、列表/表单 | 报表、图表、大屏、Excel导出 |

### Step 4: 埋点需求识别

埋点是数仓 ODS 层的重要数据源。识别并标记：
- 用户行为事件（点击、浏览、停留时长）
- 业务事件（下单、支付、退款）
- 系统事件（错误日志、性能指标）

### Step 5: 分层与引擎建议

根据需求特征建议数仓分层和查询引擎：

**数仓分层判断：**
- `ODS`：原始数据接入（埋点、业务库同步）
- `DWD`：明细数据清洗（字段标准化、数据质量）
- `DWS`：汇总数据（预聚合指标）
- `DA`：应用数据（直接支撑报表/接口）

**引擎选择建议：**
- **Hive (Tez)**：大批量历史数据处理、复杂ETL、T+1报表
- **Impala**：交互式查询、即席分析、秒级响应
- **Doris**：实时分析、高并发查询、实时大屏

**Doris 表模型选择：**
- **Aggregate Model**：预聚合指标（SUM/MAX/MIN/COUNT）
- **Unique Model**：需要 Upsert 语义的维度表
- **Duplicate Model**：明细日志、全量保留

**Doris 跨引擎关联识别：**

当目标引擎为 Doris 且数据源包含 Hive 表时，自动识别为跨引擎场景。触发信号：
- 需求涉及"实时+离线关联"、"Doris 关联 Hive"、"维度表关联"
- 目标引擎为 Doris，但所需数据存在于 Hive 表中
- 需要准实时查询但依赖离线维度或事实数据

**同步策略判断（按 Hive 源表数据量）：**
- Hive 表 **<100w 行**（维度表等）→ **Catalog 直查**：Doris SQL 中 `JOIN hive_bdsp.库.表`
- Hive 表 **≥100w 行** → **INSERT 同步**：定时将 Hive 数据写入 Doris 本地表（特大表须过滤有效数据）

**Doris 元数据获取：** Doris 表无 MCP 元数据服务。识别到 Doris 引擎时，提示用户：
> "Doris 表无元数据服务，请提供相关 Doris 表的建表 DDL 语句。"

### Step 6: 元数据补全 (可选)

当配置了 `search-hive-metadata` MCP Server 时，自动调用元数据搜索服务补全需求：

**补全流程**:
```
提取的业务术语（如"放款金额"）
        ↓
search_by_comment(term="放款金额")
        ↓
找到匹配: dwd.dwd_loan_detail.loan_amount
        ↓
补全到需求的"数据来源"和"相关字段"
```

**可用的 MCP 工具**:
- `search_table`: 按表名搜索
- `search_by_comment`: 按业务术语搜索
- `get_table_detail`: 获取表详情
- `list_columns`: 获取字段列表
- `search_existing_indicators`: 搜索指标库（Step 8.4 指标通用度评估时使用）

**MCP 不可用时降级**: 若 MCP Server 连接失败，跳过自动补全，将"数据来源"和"相关字段"标记为"待补全（MCP 不可用）"，不阻塞后续步骤。

**补全规则**:
1. 对每个需求中的指标/维度，调用 `search_by_comment` 查找对应字段
2. 搜索结果唯一时，直接补全
3. 搜索结果不唯一时，触发交互式选择
4. 搜索无结果时，标记为"待确认数据来源"

### Step 7: 数据粒度识别

在进行建模决策前，必须明确表的数据粒度。粒度决定了是否可以复用现有表。

**粒度获取优先级**:
```
1. 优先从 TBLPROPERTIES 获取
   SHOW TBLPROPERTIES table_name('granularity')
        ↓
   若返回为空或无此属性
        ↓
2. 从表注释 (table_comment) 中解析
   约定格式: [粒度：字段1，字段2，...]
   示例: [粒度：申请号，统计日期]
        ↓
   使用正则提取: \[粒度[：:]\s*([^\]]+)\]
        ↓
3. 若仍无法获取，标记为"粒度待确认"
```

**粒度匹配决策**:
- **粒度完全匹配**: 可复用现有表，建议 `ALTER TABLE ADD COLUMN`
- **粒度更细**: 需要新建表（如现有表是日粒度，需求是小时粒度）
- **粒度更粗**: 可从现有表聚合得到（如现有表是申请粒度，需求是客户粒度）
- **粒度无法确定**: 询问用户确认或联系表负责人

## Step 8: 需求文件持久化

需求清单输出后，自动写入 `docs/wip/req-{table_name}.md`，确保跨会话不丢失。详见 [persistence.md](references/persistence.md)。

### Step 8.1: 多表复杂度检测

在 Step 8 持久化完成后，分析已提取的需求，判断是否需要多表任务分解。

**检测信号**（按权重）：

| 信号 | 权重 | 示例 |
|------|------|------|
| 显式多表关键词 | HIGH | "先建中间表"、"基于X表再建Y"、"分两步" |
| 多粒度需求 | MEDIUM | 同时需要明细+汇总，或 dm+da 两层输出 |
| 依赖链关键词 | MEDIUM | "汇总后再..."、"先计算...再..."、"基于...生成..." |
| 指标数量 >10 且主题分散 | LOW | 仅建议，不自动触发 |

**判定规则**: 任一 HIGH 或 2+ MEDIUM → 判定为多表。

**执行流程**:

```
Step 8 持久化完成
    ↓
扫描需求清单，匹配检测信号
    ↓
├─ 命中 HIGH 或 2+ MEDIUM → 提示用户：
│     "检测到需求涉及多张表（{原因}），建议启用多表模式。确认？"
│     ├─ 用户确认 → 进入 Step 8.2 任务分解 → Step 8.3 公共层分析
│     └─ 用户拒绝 → 单表模式 → 检查是否触发 Step 8.4
│
├─ 无命中 或 仅 LOW → 单表模式
│     └─ 建议分层=DA（不区分大小写）? → 进入 Step 8.4 单表 dm 下沉分析
│                              否 → 流程不变，进入 Phase 2
│
└─ --plan 参数 → 跳过检测，直接进入 Step 8.2 → Step 8.3
```

### Step 8.2: 任务分解（仅多表模式触发）

将复杂需求分解为多个独立任务，每个任务对应一张目标表。

> 本节聚焦"如何从需求分解任务"。Task Registry 字段定义、状态机、DAG 执行、动态分解等完整编排规范见 [multi-table-orchestration.md](../dw-dev-workflow/references/multi-table-orchestration.md)（§7 任务分解规则、§9 职责边界）。

**分解规则**（按优先级）：粒度拆分 > 分层拆分 > 主题拆分 > 依赖推断，详见 [multi-table-orchestration.md §7](../dw-dev-workflow/references/multi-table-orchestration.md)。

**输出契约（供编排层消费）**:

1. 输出任务草案（task draft）列表，最少包含：`target_table`、`layer`、`depends_on`（可空）、`split_reason`
2. 为每个草案任务生成/更新 `docs/wip/req-{table_name}.md`
3. 在 `dw-dev-workflow` 中执行时：由编排层落盘 plan 并分配 `task_id`
4. 独立执行时：输出建议 DAG，标注为"草案"

**project_name 命名**: 从需求主题推导，snake_case 格式（如 `loan_overdue_report`、`monthly_repay_summary`）。

### Step 8.3: 公共层分析（仅多表模式，任务数 >= 2 时触发）

对每对目标表计算公共度评分（指标重叠40 + 事实表重叠30 + 维度包含20 + 粒度阶梯10）：
- **>= 40** → EXTRACT，提取公共 dm 层（维度码模式，消费方按需 JOIN 维表）
- **20-39** → SUGGEST，展示分数明细，用户选择
- **< 20** → SKIP

任务数 > 2 时使用贪心聚类合并公共层，约束公共层数 <= ceil(任务数/2)。公共层粒度 + 指标 → 输入 CASE A/B/C 决策（复用现有逻辑）。

详细评分模型、维度码模式、S5 语义守卫、聚类策略和输出格式见 [common-layer-analysis.md](references/common-layer-analysis.md)。

### Step 8.4: 单表 dm 下沉分析（单表模式 + 目标层=DA 时触发）

对 DA 层目标表的每个指标进行通用度评分（通用信号 - 专用信号）：
- **通用指标 >= 3 且无现有 dm** → SINK-NEW，新建 dm + da 两表
- **通用指标 >= 2 且有可扩列 dm** → SINK-EXTEND，扩列现有 dm + da
- **通用指标 >= 2 但计算简单且无现有 dm** → SUGGEST，提示可考虑，用户选择
- **通用指标 < 2** → SKIP，保持单表

触发下沉时升级为多表模式。详细评分模型、现有 dm 搜索和输出格式见 [dm-sink-analysis.md](references/dm-sink-analysis.md)。

## 交互式确认

遇到以下情况时，主动询问用户确认：

1. **边界模糊**: "检测到'用户列表展示累计消费金额'，这可能是后端实时计算，也可能是数仓 T+1 计算。请确认是否需要数仓开发？"
2. **时效性不明**: "需求未明确时效性要求。'订单统计'是需要实时更新还是 T+1 即可？"
3. **数据源不明**: "需求涉及'用户活跃度'指标，但未说明数据来源。是基于埋点数据还是业务库数据？"
4. **图文冲突**: "交互图显示按'日期+渠道+产品'三个维度筛选，但文本需求仅提到'按日期统计'。请确认实际需要哪些维度？"
5. **交互图信息不清晰**: "交互图中表格列头模糊/被截断，无法确认第3列是'放款金额'还是'授信金额'，请确认。"
6. **仅有交互图无文字说明**: "本需求仅从交互图提取，无对应文本描述。请确认以下理解是否正确：[列出从图中解析的指标和维度]"
7. **Excel 口径歧义**: "Excel 模板中'逾期率'列的公式为 `逾期金额/放款金额`，但文本需求描述为'逾期笔数占比'。请确认逾期率的计算口径。"
8. **Excel 字段多于文本**: "Excel 模板中包含'客户经理'、'所属支行'两列，但文本需求未提及。请确认导出是否需要这些字段？"
9. **仅有 Excel 模板无文字说明**: "本需求仅从 Excel 导出模板反推，无对应文本描述。请确认以下理解是否正确：[列出从模板中解析的字段清单和聚合方式]"
