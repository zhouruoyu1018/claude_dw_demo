# 数仓需求拆解 (DW Requirement Triage)

从混合型业务需求文档、交互图（原型图/Wireframe/Mockup）和 Excel 导出模板中精准识别数据仓库开发范畴的内容，输出结构化的数仓需求清单。

## 输入类型

本 skill 支持三类输入，可单独使用或任意组合：

| 输入类型 | 格式 | 说明 |
|---------|------|------|
| **文本文档** | PRD/BRD 文本、Markdown、复制粘贴 | 传统需求文档 |
| **交互图** | 截图(PNG/JPG)、Figma 导出图、原型图照片 | 产品提供的 UI 设计稿 |
| **Excel 导出模板** | .xlsx/.xls 文件、Excel 截图 | 产品提供的报表导出样例模板 |

> 当多类输入同时存在时，执行多源交叉验证（见 Step 2）。

## 工作流程

```
输入(文档+交互图+Excel模板) → 交互图解析 → Excel模板解析 → 结构化扫描 → 多源交叉验证 → 语义级过滤 → 埋点识别 → 分层/引擎建议 → 输出结构化需求
```

### Step 0: 交互图解析（如有）

当用户提供了交互图（截图、原型图、设计稿）时，优先执行本步骤。通过 Read 工具读取图片文件，进行视觉解析。

**解析目标**：从交互图中提取以下数仓相关元素：

| UI 元素 | 映射目标 | 提取示例 |
|---------|---------|---------|
| **筛选器/筛选条件** | 维度 | 日期选择器 → `stat_date`；渠道下拉框 → `channel` |
| **表格列头** | 指标 + 维度 | "放款金额(万元)" → 指标；"产品名称" → 维度 |
| **图表 X/Y 轴标签** | 时间粒度 + 指标 | X轴"2024-01~2024-12" → 月粒度；Y轴"逾期率%" → 指标 |
| **Tab/切换器** | 多粒度/多维度需求 | "按日｜按月｜按年" → 需支持多时间粒度 |
| **导出按钮** | 导出需求 | "导出 Excel" → 标记数据导出需求 |
| **下钻链接/跳转箭头** | 层级关系 | "点击查看明细 →" → 需要 DA 层明细表 |
| **数字卡片/KPI 区域** | 核心指标 | "今日放款 ¥1,234万" → 指标 `loan_amount` |
| **图表类型** | 分析类型 | 折线图 → 趋势分析；饼图 → 占比分析；柱状图 → 对比分析 |

**解析流程**:
```
读取交互图
    ↓
识别页面类型（报表页/大屏/列表页/详情页/表单页）
    ↓
├─ 报表页/大屏 → 高概率数仓需求，提取全部 UI 元素
├─ 列表页 → 关注筛选器和统计汇总区域
├─ 详情页 → 关注是否有统计指标区域
└─ 表单页 → 通常非数仓需求，但注意下拉选项是否来自维度表
    ↓
输出: 交互图解析结果（维度列表、指标列表、时间粒度、图表类型、下钻关系）
```

**多张交互图处理**: 如果用户提供多张交互图，逐张解析并标注页面编号，在后续需求中关联来源图。

### Step 0.1: Excel 导出模板解析（如有）

当用户提供了 Excel 导出模板（.xlsx/.xls 文件或 Excel 截图）时，执行本步骤。Excel 模板比交互图和文本更精确，因为字段名、数据格式、列顺序都是明确定义的。

**解析目标**：从 Excel 模板中提取以下数仓相关元素：

| Excel 元素 | 映射目标 | 提取示例 |
|-----------|---------|---------|
| **列头（表头行）** | 字段名 → 维度或指标 | "产品名称" → 维度；"放款金额(万元)" → 指标 |
| **数据格式** | 字段类型与精度 | 百分比(2位) → `DECIMAL(10,4)`；日期 → `DATE` |
| **示例数据** | 口径理解线索 | "2.35%" 在"逾期率"列 → 辅助判断是金额口径还是笔数口径 |
| **合计行/小计行** | 聚合维度与方式 | 按渠道小计 → `GROUP BY channel`；总计行 → 全局 SUM |
| **多 Sheet** | 多维度视角或多报表 | Sheet1"按日汇总"、Sheet2"按月汇总" → 多粒度需求 |
| **冻结列** | 维度列识别 | 左侧冻结的列通常是维度（如日期、渠道、产品） |
| **合并单元格** | 层级维度关系 | 一级分类合并多行 → 父子维度层级 |
| **条件格式/颜色标记** | 业务规则线索 | 红色标记逾期率>5% → 阈值告警需求 |
| **公式** | 衍生指标计算逻辑 | `=C2/B2` 在"逾期率"列 → `overdue_amount / loan_amount` |

**解析流程**:
```
读取 Excel 模板（文件或截图）
    ↓
识别模板类型
    ↓
├─ 明细导出模板 → 每列对应一个输出字段，确定导出粒度
├─ 汇总报表模板 → 识别维度列 + 指标列 + 聚合方式
├─ 透视表模板 → 行维度 + 列维度 + 值指标
└─ 混合模板 → 多 Sheet 分别识别
    ↓
提取: 字段清单（名称、类型、格式）、聚合方式、维度层级、数据粒度
    ↓
反推: 需要哪些源表字段、需要什么聚合逻辑、数据量级预估
```

**Excel 模板的独特价值**（文本和交互图通常无法提供的）:
- **精确字段名**: 列头就是业务方期望的输出字段名，可直接映射为表字段
- **数据格式要求**: 百分比、金额精度、日期格式 → 直接确定字段类型
- **计算公式**: 公式暴露了指标间的计算关系 → 直接转化为 ETL 逻辑
- **数据量线索**: 示例数据行数 + 日期跨度 → 预估全量数据规模
- **排序需求**: 默认排序列 → 影响 Doris 表的排序键设计

**多 Sheet 处理**: 逐个 Sheet 解析，每个 Sheet 可能对应不同的报表需求或同一报表的不同粒度视角。

### Step 1: 结构化区域扫描

扫描文档目录或大标题，优先提取以下章节：
- "统计需求"、"报表需求"、"数据分析"
- "数据埋点"、"埋点设计"
- "非功能性需求-数据部分"
- "Dashboard"、"大屏"、"BI"

### Step 2: 多源交叉验证（多类输入同时存在时）

当同时存在文本文档、交互图、Excel 模板中的两种或以上时，执行交叉验证以提升需求完整性和准确度。

**验证矩阵**:

| 场景 | 处理方式 | 确信度 |
|------|---------|--------|
| **多源一致**: 文本/交互图/Excel 模板描述吻合 | 直接确认，标记为高确信度需求 | 高 |
| **图有文无**: 交互图中有筛选器/列头，但文本未提及 | 补充到需求清单，标注"来源：交互图，文本未描述" | 中 |
| **Excel有文无**: Excel 模板有此列，但文本未提及 | 补充到需求清单，标注"来源：Excel模板，文本未描述" | 中 |
| **文有图/Excel无**: 文本描述了指标，但交互图和 Excel 模板中均无 | 保留需求，标注"仅文本描述，可能是后续迭代" | 中 |
| **多源冲突**: 不同来源对同一字段的定义不一致 | 标记为待确认项，列出冲突细节 | 低 |

**常见的"图有文无"补全场景**（产品经理容易在文档中遗漏但会画在图上）:
- 筛选器中的维度（产品经理认为"一看就懂"不写进文档）
- 表格排序方式（升序/降序、默认排序字段）
- 分页与数据量提示（"共 1,234 条"暗示明细查询需求）
- 小计/合计行（暗示需要聚合计算）

**常见的"Excel有文无"补全场景**（Excel 模板比文本更精确的地方）:
- 精确的字段名和列顺序（文本通常只描述核心指标，遗漏辅助字段）
- 数据格式要求（百分比精度、金额单位，文本很少写明）
- 计算公式暴露的指标口径（文本模糊说"逾期率"，Excel 公式明确是金额口径还是笔数口径）
- 合计/小计的聚合维度（文本说"汇总"，Excel 明确按哪些维度小计）

**Excel 模板与交互图的互补关系**:
- **交互图擅长**: 筛选条件、下钻关系、可视化方式、用户交互流程
- **Excel 模板擅长**: 精确字段定义、数据格式、计算公式、输出粒度
- 同一需求若同时有交互图和 Excel 模板，以 Excel 模板的字段定义为准（更精确），以交互图的筛选/下钻为补充

### Step 3: 语义级过滤

按段落进行 OLTP vs OLAP 特征识别：

| 特征维度 | 应用开发 (排除) | 数仓开发 (保留) |
|---------|----------------|----------------|
| **关键词** | 录入、修改、点击、跳转、校验、弹窗 | 统计、分析、汇总、监控、下钻、趋势、环比、同比 |
| **时间维度** | 实时、当前状态 | 历史快照、T+1、周期性（每日/每月） |
| **数据形态** | 单条记录、事务性、明细操作 | 聚合数据、指标、维度 |
| **输出形式** | 页面跳转、弹窗、API响应、列表/表单 | 报表、图表、大屏、Excel导出 |

### Step 4: 埋点需求识别

埋点是数仓 ODS 层的重要数据源。识别并标记：
- 用户行为事件（点击、浏览、停留时长）
- 业务事件（下单、支付、退款）
- 系统事件（错误日志、性能指标）

### Step 5: 分层与引擎建议

根据需求特征建议数仓分层和查询引擎：

**数仓分层判断：**
- `ODS`：原始数据接入（埋点、业务库同步）
- `DWD`：明细数据清洗（字段标准化、数据质量）
- `DWS`：汇总数据（预聚合指标）
- `DA`：应用数据（直接支撑报表/接口）

**引擎选择建议：**
- **Hive (Tez)**：大批量历史数据处理、复杂ETL、T+1报表
- **Impala**：交互式查询、即席分析、秒级响应
- **Doris**：实时分析、高并发查询、实时大屏

**Doris 表模型选择：**
- **Aggregate Model**：预聚合指标（SUM/MAX/MIN/COUNT）
- **Unique Model**：需要 Upsert 语义的维度表
- **Duplicate Model**：明细日志、全量保留

### Step 6: 元数据补全 (可选)

当配置了 `search-hive-metadata` MCP Server 时，自动调用元数据搜索服务补全需求：

**补全流程**:
```
提取的业务术语（如"放款金额"）
        ↓
search_by_comment(term="放款金额")
        ↓
找到匹配: dwd.dwd_loan_detail.loan_amount
        ↓
补全到需求的"数据来源"和"相关字段"
```

**可用的 MCP 工具**:
- `search_table`: 按表名搜索
- `search_by_comment`: 按业务术语搜索
- `get_table_detail`: 获取表详情
- `list_columns`: 获取字段列表

**MCP 不可用时降级**: 若 MCP Server 连接失败，跳过自动补全，将"数据来源"和"相关字段"标记为"待补全（MCP 不可用）"，不阻塞后续步骤。

**补全规则**:
1. 对每个需求中的指标/维度，调用 `search_by_comment` 查找对应字段
2. 搜索结果唯一时，直接补全
3. 搜索结果不唯一时，触发交互式选择
4. 搜索无结果时，标记为"待确认数据来源"

### Step 7: 数据粒度识别

在进行建模决策前，必须明确表的数据粒度。粒度决定了是否可以复用现有表。

**粒度获取优先级**:
```
1. 优先从 TBLPROPERTIES 获取
   SHOW TBLPROPERTIES table_name('granularity')
        ↓
   若返回为空或无此属性
        ↓
2. 从表注释 (table_comment) 中解析
   约定格式: [粒度：字段1，字段2，...]
   示例: [粒度：申请号，统计日期]
        ↓
   使用正则提取: \[粒度[：:]\s*([^\]]+)\]
        ↓
3. 若仍无法获取，标记为"粒度待确认"
```

**粒度解析示例**:
```
表注释: "贷款申请明细表，记录每笔申请的详细信息 [粒度：申请号，统计日期]"
        ↓
解析结果: ["申请号", "统计日期"]
        ↓
用于建模决策: 判断新需求的 Group By Keys 是否与现有表粒度匹配
```

**粒度匹配决策**:
- **粒度完全匹配**: 可复用现有表，建议 `ALTER TABLE ADD COLUMN`
- **粒度更细**: 需要新建表（如现有表是日粒度，需求是小时粒度）
- **粒度更粗**: 可从现有表聚合得到（如现有表是申请粒度，需求是客户粒度）
- **粒度无法确定**: 询问用户确认或联系表负责人

## 示例

**输入文档片段：**

> 用户在"我的订单"页面点击"申请退款"按钮，系统需校验订单是否超过7天，若未超过，弹出填写退款原因的弹窗。退款原因支持下拉选择，包括"质量问题"、"拍错"等。
>
> 管理后台需增加"退款分析"模块，展示每日各原因的退款单量分布，以及退款金额的周环比趋势。
>
> 运营人员需要导出过去一年的退款明细 Excel，包含用户ID、退款时间和原因。
>
> 用户点击退款按钮时需要埋点，记录用户ID、订单ID、点击时间。

**输出：**

```markdown
# 数仓需求清单

## 概览
- **文档名称**: 退款功能需求文档
- **提取时间**: 2024-01-15
- **需求数量**: 3 项

## 需求列表

### 1. 退款分析报表
- **原文描述**: 管理后台需增加"退款分析"模块，展示每日各原因的退款单量分布，以及退款金额的周环比趋势
- **需求类型**: 聚合统计
- **涉及指标**: 退款单量, 退款金额, 周环比
- **涉及维度**: 日期, 退款原因
- **时间粒度**: 日
- **建议分层**: DA
- **建议引擎**: Impala (交互式查询) 或 Doris (如需实时)
- **优先级评估**: 高
- **备注**: 需要 DWS 层预聚合退款汇总表

### 2. 退款明细导出
- **原文描述**: 运营人员需要导出过去一年的退款明细Excel，包含用户ID、退款时间和原因
- **需求类型**: 明细查询
- **涉及指标**: 无（明细导出）
- **涉及维度**: 用户ID, 退款时间, 退款原因
- **时间粒度**: 年
- **建议分层**: DWD
- **建议引擎**: Hive (大数据量导出)
- **优先级评估**: 中
- **备注**: 一年数据量较大，建议分批导出

### 3. 退款按钮点击埋点
```

## 交互式确认

遇到以下情况时，主动询问用户确认：

1. **边界模糊**: "检测到'用户列表展示累计消费金额'，这可能是后端实时计算，也可能是数仓 T+1 计算。请确认是否需要数仓开发？"

2. **时效性不明**: "需求未明确时效性要求。'订单统计'是需要实时更新还是 T+1 即可？"

3. **数据源不明**: "需求涉及'用户活跃度'指标，但未说明数据来源。是基于埋点数据还是业务库数据？"

4. **图文冲突**: "交互图显示按'日期+渠道+产品'三个维度筛选，但文本需求仅提到'按日期统计'。请确认实际需要哪些维度？"

5. **交互图信息不清晰**: "交互图中表格列头模糊/被截断，无法确认第3列是'放款金额'还是'授信金额'，请确认。"

6. **仅有交互图无文字说明**: "本需求仅从交互图提取，无对应文本描述。请确认以下理解是否正确：[列出从图中解析的指标和维度]"

7. **Excel 口径歧义**: "Excel 模板中'逾期率'列的公式为 `逾期金额/放款金额`，但文本需求描述为'逾期笔数占比'。请确认逾期率的计算口径。"

8. **Excel 字段多于文本**: "Excel 模板中包含'客户经理'、'所属支行'两列，但文本需求未提及。请确认导出是否需要这些字段？"

9. **仅有 Excel 模板无文字说明**: "本需求仅从 Excel 导出模板反推，无对应文本描述。请确认以下理解是否正确：[列出从模板中解析的字段清单和聚合方式]"
