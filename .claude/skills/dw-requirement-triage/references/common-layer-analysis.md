# Step 8.3: 公共层分析

在 Step 8.2 任务分解完成后，分析各任务间的指标、源表和维度重叠度，判断是否需要提取公共 dm 层。

**触发条件**: Step 8.2 输出任务数 >= 2。

## 覆盖场景

| # | 场景 | 示例 |
|---|------|------|
| S1 | 同指标不同粒度 | 放款日报(日+产品) + 放款月报(月+产品)，"放款金额"两边都要 |
| S2 | 同指标不同维度组合 | 放款产品报(按产品) + 放款渠道报(按渠道)，"放款金额/笔数"都要 |
| S3 | 同源表不同输出 | 两张报表都 JOIN dwd_loan_dtl + dim_product，但聚合维度和输出指标不同 |
| S4 | 逐级聚合（粒度阶梯） | 明细→日→月，月表可从日表聚合而非重新扫描明细 |
| S5 | 同源不同过滤 | 同一明细表，A报表取"已放款"，B报表取"已签约"，共用 JOIN 但 WHERE 不同 |

## 评分模型

对每对目标表 (Ti, Tj) 计算公共度分数（满分 100）：

```
指标重叠分 (满分 40):
  score_metric = |Ti指标 ∩ Tj指标| / min(|Ti指标|, |Tj指标|) x 40
  -- 覆盖: S1, S2, S5

源表重叠分 (满分 30):
  score_source = |Ti事实表 ∩ Tj事实表| / min(|Ti事实表|, |Tj事实表|) x 30
  -- 覆盖: S3
  -- 注意: 仅计算事实表交集，维表不参与评分（见"维度码模式"）

维度包含分 (满分 20):
  Ti维度 ⊆ Tj维度 或 Tj维度 ⊆ Ti维度 → 20
  |Ti维度 ∩ Tj维度| >= 1 但无包含关系 → 10
  无交集 → 0
  -- 覆盖: S1, S2

粒度阶梯分 (满分 10):
  Ti粒度可由Tj聚合得到，或反向 → 10
  否则 → 0
  -- 覆盖: S4
```

## 判定与动作

```
总分 >= 40 → EXTRACT: 建议提取公共 dm 层
  ├─ 计算公共层粒度（见"粒度计算规则"）
  ├─ 计算公共层指标（见"指标合并规则"）
  ├─ 计算公共层源表（见"源表选择规则"）
  ├─ 多对表时执行贪心聚类（见"多表聚类策略"）
  ├─ 公共层粒度 + 指标 → 输入 generate-standard-ddl Step 2 的 CASE A/B/C 决策
  │   （复用现有建模决策逻辑，不新增 CASE）
  ├─ 输出公共层草案（target_table, layer=dm, 粒度, 指标, 源表）
  ├─ 新增公共层任务，修改 Ti/Tj 的 depends_on 指向公共层
  └─ 展示优化后 DAG，等待用户确认

总分 20-39 → SUGGEST: 提示可考虑提取，展示分数明细
  └─ 用户确认提取 → 同 EXTRACT 流程
  └─ 用户拒绝 → 保持原有独立任务

总分 < 20 → SKIP: 不建议提取，各表独立开发
```

## 维度码模式（默认策略）

公共层默认采用**星型模型维度码模式**：只保留事实表级粒度 + 维度外键码（如 `channel_id`, `org_id`, `product_id`），不做维表 JOIN 展开。

**好处**:
- 公共层源表只需事实表交集，不用拉维表，避免粒度与源表的冲突
- 维度码天然是最细粒度，消费方 GROUP BY 聚合到任意粒度
- 数据量可控，行数 = 事实表行数

**消费方用法**:
```sql
-- Ti: 按渠道统计
SELECT c.channel_name, SUM(common.loan_amount)
FROM common_layer common
JOIN dim_channel c ON common.channel_id = c.channel_id
GROUP BY c.channel_name;

-- Tj: 按机构统计
SELECT o.org_name, SUM(common.loan_amount)
FROM common_layer common
JOIN dim_org o ON common.org_id = o.org_id
GROUP BY o.org_name;
```

## 源表选择规则

```
公共层源表 = 事实表交集 ∪ 公共粒度所需的维表（仅当维度码不在事实表中时）
```

大多数情况下，维度外键码已在事实表中（如 `channel_id`, `org_id`），公共层无需 JOIN 维表。仅当维度码不存在于事实表中（需要通过维表关联获取）时，才将该维表纳入公共层源表。

## 指标合并规则

公共层指标 = Ti指标 ∩ Tj指标（名称匹配），但需排除以下情况：

### S5 场景：同名指标不同语义

当同名指标的 WHERE 条件不同时（如 A 取"已放款"、B 取"已签约"），按过滤条件类型分流：

```
├─ 过滤条件为枚举维度（status IN ('放款','签约')，值域小且可枚举）
│   → 公共层保留全量，新增状态列（如 loan_status），消费方各自 WHERE
│   → 指标保留在公共指标中
│
├─ 过滤条件为高基数/复杂表达式（如 amount > threshold、复杂日期计算）
│   → 该指标从公共指标中排除，各表独立计算
│
└─ 无法判断 → 标记为"待确认"，询问用户
```

## 粒度计算规则

| 场景 | 公共层粒度 |
|------|----------|
| Ti(日+产品) Tj(月+产品) | 日+产品（取最细） |
| Ti(产品+渠道) Tj(产品+机构) | 产品+渠道+机构（取维度码并集） |
| Ti(日+产品) Tj(日+产品+渠道) | 日+产品+渠道（细粒度已包含粗粒度） |

原则: 公共层粒度必须足够细以支撑所有消费方，消费方通过 GROUP BY 聚合到自己的粒度。在维度码模式下，粒度通常为事实表的自然粒度（如申请号级别），消费方按需聚合。

## 多表聚类策略

当任务数 > 2 时，pairwise 评分可能产生多个公共层。使用**贪心聚类**合并：

```
1. 对所有表对按公共度分数降序排列
2. 取最高分的表对 (Ti, Tj)，创建公共层 C1
   C1.粒度 = Ti维度码 ∪ Tj维度码 中取最细
   C1.指标 = Ti指标 ∩ Tj指标
   C1.源表 = Ti事实表 ∩ Tj事实表 (+ 必要维表)
3. 遍历剩余任务 Tk:
   ├─ Tk 的指标是 C1.指标 的子集 且 Tk 的粒度可从 C1 聚合得到
   │   → 将 Tk 并入 C1 的消费方
   └─ 否则 → 创建新公共层 C2（或 Tk 保持独立）
4. 约束: 公共层数 <= ceil(任务数 / 2)
   若公共层数 > ceil(任务数 / 2)，放弃分数最低的公共层，其消费方回退为独立任务
```

**示例**（3 个任务）:
```
A↔B=80, A↔C=60, B↔C=50
  → C1 基于 A↔B 创建
  → C 的指标 ⊆ C1.指标? 粒度可聚合?
    → 是 → C 并入 C1。结果: 1 个公共层服务 A、B、C
    → 否 → C 独立。结果: 1 个公共层服务 A、B + C 独立
```
