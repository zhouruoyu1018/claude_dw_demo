# Step 2.5 逻辑流程确认（Logic Plan Review）

本文件定义 `generate-etl-sql` 在复杂场景下的“先逻辑、后写 SQL”门禁。

## 触发条件

| 加工模式 | 是否触发 | 说明 |
|---------|---------|------|
| 简单聚合 | 跳过 | 逻辑直观，直接生成 SQL |
| 增量加载 | 跳过 | 模式固定 |
| 多表关联 | 可选 | 2 表以内可跳过，3+ 表建议触发 |
| 窗口计算 | 触发 | 逻辑复杂 |
| 分组集 | 触发 | 维度组合复杂 |
| 混合模式 | 必须 | 包含多类复杂逻辑 |

## 输出要求（Logic Plan）

在写 SQL 前先输出逻辑流程，至少包含：

1. 目标与口径
2. 数据源与粒度
3. 分步骤处理流（每一步对应一个 CTE）
4. 每一步的输入、过滤、聚合、输出字段和粒度变化

建议结构：

- `base_*`：基础过滤
- `agg_*`：聚合层
- `win_*`：窗口层
- `final`：结果组装

## 自检清单

### 固定检查项（每次必检）

1. 是否存在 N:N JOIN 膨胀风险
2. 分母为 0 是否处理
3. NULL 传播是否处理
4. 粒度是否逐步收敛到目标粒度
5. JOIN 条件是否完整（含分区过滤/无分区说明）
6. 窗口函数 PARTITION/ORDER 与边界是否正确
7. 业务范围覆盖度是否一致
8. JOIN 键近名歧义是否排除
9. JOIN 键语义类型是否匹配

### 动态检查项（从 pitfalls.md 加载）

按本次涉及源表匹配已知问题并输出动态自检结果；无匹配时显式输出“无”。

## 重点校验规则

### 业务范围覆盖度（固定项 7）

当需求包含明确范围（产品/渠道/地区等）时：

1. 提取需求范围
2. 读取源表实际枚举值
3. 交叉比对（覆盖/缺口/多余）
4. 输出覆盖度报告

发现缺口时必须中断并询问用户，不可静默忽略。

### JOIN 键消歧（固定项 8/9）

对每个 JOIN 键执行：

1. 近名字段检测（如 `of_department_psid` vs `orphan_of_department_psid`）
2. 语义类型匹配（PSID/编码/ID）

若存在歧义或语义不匹配，必须中断并询问用户确认。

## 问题回写

当自检发现真实问题或用户确认问题成立时，回写 `pitfalls.md`：

- 源表特性问题 → “源表已知特性”
- ETL 编写错误 → “ETL 常见错误”
- DQC 捕获真实缺陷 → “DQC 捕获的真实缺陷”

## 用户确认门禁

输出逻辑流和自检结果后，需用户选择：

- 确认并生成 SQL
- 修改指定步骤
- 补充约束条件

## 可跳过条件

满足任一条件可跳过 Step 2.5：

1. 用户明确要求直接生成 SQL
2. 简单聚合模式（单表 + 单层 GROUP BY）
3. 用户已提供完整映射模板

跳过时需在脚本头注释标记“逻辑流程已跳过及原因”。
