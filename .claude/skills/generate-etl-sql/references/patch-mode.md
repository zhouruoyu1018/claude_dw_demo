# Patch 模式详细步骤

> 本文件是 `generate-etl-sql` Patch 模式 (--mode=patch) 的 Step P0~P3 详细规则。
> SKILL.md 中保留骨架和流程图，具体步骤引用本文件。

---

## Step P0: 加载与解析现有脚本

### P0.1 加载现有脚本

**输入来源优先级**：
1. **工作流传入**: dw-dev-workflow Phase 4 自动传入 ETL 文件路径
2. **用户指定**: 用户提供文件路径（如 `sql/hive/etl/dmm_sac_cm3_migr_org_etl.sql`）
3. **约定搜索**: 根据目标表名在 `sql/hive/etl/` 下查找 `{table_name}_etl.sql`
4. **未找到**: 询问用户提供路径；用户无现有脚本 → 降级为 incremental 模式（新建 ETL）

### P0.2 解析脚本结构

读取现有脚本后，解析并提取：
- **头部注释**: 功能描述、源表列表、修改记录
- **SET 参数**: 引擎配置
- **CTE 结构**: 每个 CTE 的名称、作用、涉及的源表和字段
- **最终 SELECT**: 字段列表及其来源 CTE
- **JOIN 关系**: 表间关联键
- **分区策略**: 静态/动态分区

### P0.3 加载变更需求

从以下来源获取变更描述：
1. **同会话上下文**: dw-dev-workflow 传入的 ALTER TABLE DDL + 需求描述
2. **需求文件**: `docs/wip/req-{table_name}.md` 中的变更记录
3. **用户直接描述**: 用户在对话中说明变更内容

**变更类型分类**:

| 类型 | 说明 | 示例 |
|------|------|------|
| ADD_FIELD | 新增字段 | "新增当日放款笔数字段" |
| MODIFY_LOGIC | 修改字段计算逻辑 | "放款金额改为只统计成功状态" |
| ADD_SOURCE | 新增源表关联 | "需要关联产品维度表获取产品名称" |
| REMOVE_FIELD | 移除字段 | "移除已废弃的xxx字段" |

---

## Step P1: 变更影响分析

对每个变更项，分析：
1. **影响范围**: 哪些 CTE 需要修改、是否需要新建 CTE
2. **源表依赖**: 是否需要新增 JOIN（新源表需先通过 `search-hive-metadata` 获取 Schema）
3. **字段传递链**: 从源字段 → CTE → 最终 SELECT 的完整路径
4. **副作用检查**: 修改是否影响现有字段的正确性（如改了 GROUP BY 键）

**输出影响分析摘要**，供用户确认后继续。

---

## Step P2: 变更方案确认

输出变更计划表：

```
═══════════════════════════════════════════════════════════════
 变更计划 (Patch Plan) - 请确认
═══════════════════════════════════════════════════════════════

┌─────────────┬──────────────┬──────────────────┬──────────┐
│ 变更项       │ 影响位置      │ 具体操作          │ 风险     │
├─────────────┼──────────────┼──────────────────┼──────────┤
│ 新增 xxx_cnt │ CTE: agg_xxx │ 添加 COUNT(...) │ 无       │
│             │ 最终 SELECT  │ 添加字段         │          │
│ 修改 yyy_amt │ CTE: base    │ 增加 WHERE 条件  │ 影响下游 │
└─────────────┴──────────────┴──────────────────┴──────────┘

确认方式:
(A) 全部正确，继续修改
(B) 调整第 [N] 项的修改方案
(C) 取消变更

═══════════════════════════════════════════════════════════════
```

用户确认后进入 Step P3。

## Step P3: 应用变更

### P3.1 修改 CTE（如需要）
- 在现有 CTE 中添加字段或修改逻辑
- 如需新增 CTE，在合适位置插入（保持 CTE 依赖顺序）
- 保持现有 CTE 命名规范和注释风格

### P3.2 修改最终 SELECT
- 在对应分组位置（维度→布尔→指标）插入新字段
- 保持原有字段顺序不变
- 新增字段添加行内注释

### P3.3 修改 JOIN（如需要）
- 新增 LEFT JOIN 并添加注释说明
- 确保分区字段带上（避免全表扫描）

### P3.4 更新头部注释
- 在修改记录中追加 changelog：`-- {YYYY-MM-DD} {author} {变更描述}`
- 如新增源表，更新源表列表
- 如有新增指标入库，更新指标注释
- 更新核心逻辑摘要（如 CTE 结构有变化）

### P3.5 输出完整脚本
- 输出修改后的完整 ETL 脚本（非 diff）
- 文件路径与原文件相同（覆盖）
- 如有 `{table_name}_init.sql` 且变更影响 init 逻辑 → 同步更新

### P3.6 需求变更即时同步

> 复用 [input-analysis.md Step 0.6](input-analysis.md) 的同步规则，patch 模式同样适用。

Patch 模式虽跳过 Step 0~3，但在 P0~P3 任何阶段发生需求变更时，**必须立即同步更新** `docs/wip/req-{table_name}.md` 对应章节：

| 变更类型 | 更新目标章节 |
|---------|------------|
| 新增/移除源表 | `## 源表` + `## 关键源字段` |
| 调整指标口径 | `## 指标清单` 对应行 |
| 新增/删除指标 | `## 指标清单` |
| 修改维度 | `## 维度` |
| 修改字段映射 | `## 字段映射（已确认）` |

- 变更发生时**当场更新**，不等到流程结束
- 更新后标注 `(Phase 4 patch 补充, {日期})`

### P3.7 完成后流转
- Step P3 完成后，进入 **Step 4**（优化与审查），复用已有逻辑
- 再进入 **Step 5/6**（指标/血缘注册），但仅处理增量部分
